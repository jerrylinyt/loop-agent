# 🔁 RULE — 單任務收斂協定（Per-Task Convergence）

> **唯讀框架規則**。解決「單次信不過」:同一任務由不同輪次 agent **獨立重做**,連續 `converge_threshold` 次一致才定稿。
> 適用於**所有不信任單次結果的分析/設計任務**。Init 類任務(建 repo、盤點)不需收斂,做完即 CONVERGED。
> 有客觀把關的任務(如有編譯器/測試的實作任務)以 build/test/行為對照為收斂依據,不需多次主觀重推。
> 門檻值來自 `loop.config.yaml`(`phases[i].converge_threshold` 或 `extra_thresholds`)。
>
> 🚨 強制約束(收斂類別不准執行時自判):每個任務的收斂類別——「需多輪收斂」/「Init 免收斂」/
>    「有客觀把關(build/test)」——【必須】由 phase 規格(plan/config)**預先宣告**,執行輪只能照宣告走。
>    ❌ 嚴禁:執行 agent 臨場把一個「需收斂」的主觀分析任務,自稱「這基本上是盤點/已有把關」就一輪定稿
>    CONVERGED 來跳過獨立重驗。規格未宣告或有疑義 → 一律從嚴歸「需多輪收斂」。

## 任務生命週期與狀態
```
TODO ──初稿──▶ DRAFTED(conv=0)
                   │
       ┌───────────┴────────────┐
       │ 獨立重驗一致            │ 獨立重驗發現實質差異
       ▼                        ▼
  conv += 1                修正文件 + conv=0（維持 DRAFTED）+ 寫修正記錄
       │
       ▼
  conv >= threshold ──▶ CONVERGED（=定稿）
```
- **Status 值**: `TODO` / `DRAFTED` / `CONVERGED` / `NEEDS_REVISION` / `FROZEN`
- 狀態表用 `Conv` 欄記錄目前 converge_count（如 `1/2`）。
- CONVERGED 等同「已收斂定稿」。

## 獨立重驗程序（重點:先重推導,再對比,避免橡皮圖章）
```
RE-VERIFY（當任務 status==DRAFTED 且 conv<threshold）:
  1. ❗先「不要看」既有產出文件。直接從原始輸入重新推導一份獨立結果。
     🚨 【必須】把這份重推稿寫成 scratch 獨立驗證檔(如 <outputs>/.reverify/<task>-R###.md),
        使「確實做過獨立重推」可被事後稽核。❌ 嚴禁跳過重推、直接讀既有產出就宣稱一致。
     🚨 防偽獨立(驗證檔存在 ≠ 真的獨立):重推稿每項【必須】標原始輸入來源(檔:行),
        證明結論源自原始輸入而非抄既有產出;❌ 嚴禁在重推稿中引用/貼上既有產出檔的內容或路徑。
        無來源行號的重推稿視同未獨立重推,本次重驗無效、不得 +1。
  2. 再打開既有文件,逐項對比「我重推的」與「文件記錄的」。
  3. 把差異分類:
       · 實質差異 = 改變意義(漏一項、判斷錯、欄位/觸發條件錯、分支漏)
       · 表面差異 = 用字/排版不同,意義相同
     🚨 從嚴預設:任何「無法 100% 確定只是表面」的差異,一律歸類為【實質差異】。
        ❌ 嚴禁為了讓 conv 繼續累積,把實質差異低報成表面差異——有疑義就是實質。
  4. 有「實質差異」:以正確版本更新文件 → conv = 0(重新計數)→ 寫修正記錄;
     若差異源於規格矛盾/缺檔/無法解析的邏輯 → 【必須】開 Issue(見 issues.md),不得自行腦補解決。
  5. 完全沒有實質差異:conv += 1 → 日誌記「RE-VERIFY 一致 (conv=N/threshold)」,
     並在日誌標注本輪重推稿驗證檔路徑(無驗證檔 = 本次重驗無效,不得 +1)。
  6. conv >= threshold → status = CONVERGED。
```

> 🚨 **機械鐵則(產出有改 = 必歸零,可被 commit diff 客觀稽核)**:
> 「conv += 1」這條路**只在本輪完全沒動到正式產出檔時**才合法——重驗一致的那輪,你只會新增 scratch
> 重推稿(`.reverify/`),不會改正式產出。一旦本輪 commit 的 diff 動到了該任務的**正式產出檔**,就代表
> 「上一稿不是最終稿」(有得改 = 有實質差異),該任務**必須 conv 歸零、且本輪不得標 CONVERGED**,
> 走第 4 步(修正 + conv=0 + 寫修正記錄),不得走第 5 步 +1。
> ❌ 嚴禁:同一輪既改了產出、又 conv += 1 或直接標 CONVERGED(自稱「只是順手潤一下、不算實質差異」)。
> 有改就是有差異——歸零重數。此鐵則由 git-review-gate §2-13 獨立把關,違者 REVERT。
> (排除:scratch 證據檔 `.reverify/.enum/.validate`、Issue 檔、CONTROL 狀態欄位不算「正式產出」。)

## 收斂鐵則 — 獨立性
- 每次重驗【必須】由「**與產出該稿不同的另一輪 agent(新 context)**」做,獨立性才有意義(同一輪不得既產稿又自驗)。
  🚨 **強制約束**:同一輪【只能做一次】重驗(conv 最多 +1)。❌ 嚴禁在同一輪內對同一任務連做兩次以上重驗來「一次湊滿收斂門檻」——那會讓「連續 N 次獨立確認」退化成同一盲點的自我蓋章。第二次重驗必須等下一輪(外部重新喚醒的新 process、新 context)。
- **先重推/重列舉,再跟既有比對**——不可先讀既有結果再說「看起來沒問題」(橡皮圖章,同一盲點被一再確認)。
- 任一次發現實質差異 → 修正 + 收斂計數**歸零重數**。保證「連續 N 次」是對已穩定產物的 N 次獨立確認。

## 三層收斂（按任務風險選用）
| 層級 | 防什麼 | 收斂判準 | 典型門檻 |
|------|--------|----------|------|
| **單任務收斂** | 單支結果腦補/判斷錯 | 獨立重推一份 → 與既有比對,連續 N 次一致 | 3~   |
| **集合穩定性收斂** | 大範圍**漏看** | 每輪獨立重新「列舉」項目,連續 N 輪集合不再增減 | 3~   |
| **全量驗證收斂** | 跨檔/模組**不一致** | 對整個產物全量驗證,連續 N 次 PASS | 10   |

> 集合穩定性收斂的細節見 `completeness.md`(大範圍防漏)。全量驗證收斂用於 phase 收尾,門檻 = `config.stop_condition.final_phase_pass_gte`。

> ⚠️ **生成式拆解的弱訊號警告**：集合穩定性收斂對「生成式拆解」（如漸進拆解樹的每節點子項提議）是比機械列舉更弱的訊號——「連續 N 輪沒長新子節點」可能是真的拆完，也可能是弱模型沒梗了。**故程式收斂只保證「穩定」，不保證「正確」；正確性由人類 gate 承接**。兩者在程式碼與文件中不可混為一談。

## 樹形節點收斂協定（Tree Node Convergence Protocol）
在樹形漸進拆解中，收斂的邏輯分兩個層次：
1. **單一節點拆解收斂**：
   - 節點處於 `PENDING` 狀態時，每輪重新提議其子節點。
   - 當子節點提案（`proposed_children` 集合）連續 `converge_threshold` 輪不增、不減、不改時，判定該節點的拆解「集合穩定性收斂」。
   - 一旦子節點提案發生任何實質變動（修改名稱、新增/刪除子項），該節點的收斂計數 `conv` 立即歸零。
2. **全樹規劃期收斂**：
   - 當樹上所有 Pending 節點均完成拆解（達到收斂，狀態轉為 `DECOMPOSED`），且無其他待拆解節點時，整棵樹即達到「全局穩定收斂」（規劃期結束）。
3. **人類 Gate 判定**：
   - 程式判定收斂僅代表結構「穩定」，不代表「正確」。
   - 規劃期收斂後，必須暫停並交由人類進行 **Gate 驗收**。人類點頭批准或對局部子樹進行打回重拆。

## 注意
- 重驗發現實質差異 → 不只改文件,還要檢查有沒有其他文件引用到舊的錯誤值(雙向 traceability 連動更新)。
- 下游任務若依賴某任務,【必須】等它 CONVERGED 再開始;❌ 嚴禁在上游未穩(未 CONVERGED)時提早動下游——上游一改,下游全部得跟著歸零重做,污染收斂。
- **Context 防爆**:重驗的「對比」是比對**項目身分清單**,不是把兩份全文同時塞進 context(見 context-budget.md)。
