# 🛟 RULE — 震盪偵測、二維調度與升級協定（防卡死與失控）

> **唯讀框架規則**。解決「改 A 壞 B、改 B 壞 A」的死循環。
> **偵測由外部 loop 引擎做**（最為客觀，避免 agent 在局中自盲）。
> 本規則涵蓋：**三種卡住偵測**、**三模型二維調度（角色預設 + 升級階梯）** 以及 **授權紅線**。

## A. 三種「卡住」與偵測訊號
| 卡法 | 現象 | 偵測訊號 |
|------|------|----------|
| **不收斂** | 同一問題一直修不好 | `rounds_since_progress >= stall_threshold`（連續客觀驗收失敗輪沒進展，不論推進/驗證模式） |
| **震盪 A↔B** | 每輪都「修好一個」卻原地繞 | 最近 `osc_window` 輪的「失敗指紋」只在 ≤`osc_distinct_max` 種間循環，且有重複 |
| **無活動(空轉)** | agent 連續多輪沒提交 | 「活動簽章」連續 `stall_threshold` 輪不變 |

**失敗指紋（fingerprint）** = `sha1( 排序(last_round_fail_tasks) + "|" + 排序(本輪 git 改動檔案) )`
- **any 客觀驗收失敗都推進指紋歷史**:只要本輪實際跑了 build/test/編譯器/全量驗證/整合驗證且失敗,不論 `last_round_mode` 是「推進」或「驗證」,都必須回填 `last_round_result=FAIL` 與 `last_round_fail_tasks`,並推進失敗指紋歷史。
  🚨 強制約束:`NA` 只允許用於「本輪沒有執行任何客觀驗收」的純推進輪。❌ 嚴禁:驗收失敗後把本輪寫成 `NA`、空白、或「推進中」來規避震盪偵測。

## B. 三模型二維調度（三模型角色 × 升級階梯）
引擎使用三種模型：`fast` (快速模型), `normal` (普通模型), `thinking` (思考模型)，並在兩個維度上進行調度。

### 1. 角色維度（順風時，按任務類型指派預設模型）
- **規劃（Plan / Phase Gate）**：預設使用 `thinking`（需要全景視野與精密推理）。
- **審查與驗證（Review / Validation）**：預設使用 `normal`（獨立重推與比對）。
- **任務執行（Task execution）**：預設使用 `fast`（短 context，以快速執行與回饋為主）。

### 2. 升級維度（逆風時，卡住時沿階梯向上爬升）
當偵測到卡住（不收斂或震盪）時，工作模型沿階梯升級：
```
[ 任務預設: fast ] ──(卡住)──▶ [ normal ] ──(再卡)──▶ [ thinking ] ──(再卡)──▶ [ Lv2 人類 ]
[ 規劃預設: thinking ] ──────────(卡住)──────────▶ [ Lv2 人類 ]
```
- **降回預設**：若出現 `PASS` 或實質進展，模型立即降回該任務類型的**角色預設**模型，以節省成本。
- **Lv2 人類接管**：
  - 🚨 強制約束(凍結/交人是逃生閥,不是偷懶出口):agent 動 `FROZEN`/`human_required` 前,
    `state.json`【必須】已由**引擎**判定卡死——`stuck_level==2`(引擎設),或 `rounds_since_progress>=stall_threshold`。
    震盪偵測「由外部 loop 引擎做(最客觀)」,agent 不得自行認定卡死。
    ❌ 嚴禁:在一個只是「難、但引擎沒判卡死」的任務上自抬 `stuck_level=2`、自行凍結任務,
    把硬工作提早甩給人類。沒有引擎卡死訊號 → 就是繼續做,不是凍結。
  - 開啟一個 `BLOCKING` 狀態的 Issue，清晰記錄衝突根因；🚨【必須】在 Issue 內引用觸發的機械訊號
    (失敗指紋環內容 / `rounds_since_progress` 數值),否則此凍結視為無效。
  - 將涉及的互卡任務改為 `FROZEN`，跳過不執行。
  - 當所有非凍結任務均無法執行時，停下交人類處理 (`human_required=true`)。
  - 🚨 設定 `human_required=true`、開完 BLOCKING Issue、凍結互卡任務後，該 agent 必須【立即停止輸出、結束本 process】，把控制權交還引擎等待人類裁決。❌ 嚴禁在卡死/升級情境下還自行續推下一輪或反覆重試燒模型。

## C. 授權紅線
程式只被授權做「可機器判讀」的決定（如收斂計數、輪數等）。
凡牽涉 **價值判斷** 的決定，一律在邊界之外，程式不得擅自代判或放寬：
1. **「規格本身有矛盾」不該由程式判**：程式分不清「模型能力不夠」與「兩條需求本質衝突」，只需開 Issue 記錄現象，不可自行妥協或修改規格。
2. **「規劃書設計得對不對」不該由程式判**：程式只能驗證「結構穩定」（集合收斂），正確性必須由人類 Gate 承接。
