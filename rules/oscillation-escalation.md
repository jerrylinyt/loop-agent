# 🛟 RULE — 震盪偵測、二維調度與硬 Breaker 協定（防卡死與失控）

> **唯讀框架規則**。解決「改 A 壞 B、改 B 壞 A」的死循環，以及樹形結構下的失控增長與垂直震盪。
> **偵測由外部 loop 引擎做**（最為客觀，避免 agent 在局中自盲）。
> 本規則涵蓋：**三種卡住偵測**、**三模型二維調度（角色預設 + 升級階梯）**、**三條硬 Breaker** 以及 **授權紅線**。

## A. 三種「卡住」與偵測訊號
| 卡法 | 現象 | 偵測訊號 |
|------|------|----------|
| **不收斂** | 同一問題一直修不好 | `rounds_since_progress >= stall_threshold`（連續失敗驗證輪沒進展） |
| **震盪 A↔B** | 每輪都「修好一個」卻原地繞 | 最近 `osc_window` 輪的「失敗指紋」只在 ≤`osc_distinct_max` 種間循環，且有重複 |
| **無活動(空轉)** | agent 連續多輪沒提交 | 「活動簽章」連續 `stall_threshold` 輪不變 |

**失敗指紋（fingerprint）** = `sha1( 排序(last_round_fail_tasks) + "|" + 排序(本輪 git 改動檔案) )`
- **任何客觀驗收失敗都推進指紋歷史**:只要本輪實際跑了 build/test/編譯器/全量驗證/整合驗證且失敗,不論 `last_round_mode` 是「推進」或「驗證」,都必須回填 `last_round_result=FAIL` 與 `last_round_fail_tasks`,並推進失敗指紋歷史。
  🚨 強制約束:`NA` 只允許用於「本輪沒有執行任何客觀驗收」的純推進輪。❌ 嚴禁:驗收失敗後把本輪寫成 `NA`、空白、或「推進中」來規避震盪偵測。

## B. 三模型二維調度（三模型角色 × 升級階梯）
引擎使用三種模型：`fast` (快速模型), `normal` (普通模型), `thinking` (思考模型)，並在兩個維度上進行調度。

### 1. 角色維度（順風時，按任務類型指派預設模型）
- **規劃與拆解（Root / Intermediate Nodes）**：預設使用 `thinking`（需要全景視野與精密推理）。
- **審查與整合驗證（Review / Integration）**：預設使用 `normal`（獨立重推與比對）。
- **葉子任務執行（Leaf execution）**：預設使用 `fast`（短 context，以快速執行與回饋為主）。

### 2. 升級維度（逆風時，卡住時沿階梯向上爬升）
當偵測到卡住（不收斂或震盪）時，工作模型沿階梯升級：
```
[ 葉子預設: fast ] ──(卡住)──▶ [ normal ] ──(再卡)──▶ [ thinking ] ──(再卡)──▶ [ Lv2 人類 ]
[ 整合預設: normal ] ──────(卡住)──────▶ [ thinking ] ──────(再卡)──────▶ [ Lv2 人類 ]
[ 規劃預設: thinking ] ──────────(卡住)──────────▶ [ Lv2 人類 ]
```
- **降回預設**：若出現 `PASS` 或實質進展，模型立即降回該任務類型的**角色預設**模型，以節省成本。
- **Lv2 人類接管**：
  - 🚨 強制約束(凍結/交人是逃生閥,不是偷懶出口):agent 動 `FROZEN`/`human_required` 前,
    CONTROL【必須】已由**引擎**判定卡死——`stuck_level==2`(引擎設),或 `rounds_since_progress>=stall_threshold`。
    震盪偵測「由外部 loop 引擎做(最客觀)」,agent 不得自行認定卡死。
    ❌ 嚴禁:在一個只是「難、但引擎沒判卡死」的任務上自抬 `stuck_level=2`、自行凍結任務,
    把硬工作提早甩給人類。沒有引擎卡死訊號 → 就是繼續做,不是凍結。
  - 開啟一個 `BLOCKING` 狀態的 Issue，清晰記錄衝突根因；🚨【必須】在 Issue 內引用觸發的機械訊號
    (失敗指紋環內容 / `rounds_since_progress` 數值 / 撞線的 breaker 名稱),否則此凍結視為無效。
  - 將涉及的互卡任務改為 `FROZEN`，跳過不執行。
  - 當所有非凍結任務均無法執行時，停下交人類處理 (`human_required=true`)。
  - 🚨 設定 `human_required=true`、開完 BLOCKING Issue、凍結互卡任務後，該 agent 必須【立即停止輸出、結束本 process】，把控制權交還引擎等待人類裁決。❌ 嚴禁在卡死/升級情境下還自行續推下一輪或反覆重試燒模型。

## C. 三條硬 BREAKER（撞線即凍結交人，程式不准自我放寬）
為防止樹形結構無限增長或跨層級垂直震盪，引擎設立了三條不可逾越的硬 BREAKER。**一旦撞線，立即停止運作、凍結任務、移交人類。程式絕無權限自動放寬 config 旋鈕。**

| Breaker | 攔什麼失控 | config 旋鈕 | 偵測點 | 撞線代表 |
|---------|-----------|-------------|--------|----------|
| **max_depth** | 樹拆解太深（無限遞迴） | `breaker.max_depth` (預設 5) | 規劃期，每次子節點定案時 | 最小單位定義錯誤，或規格本質有衝突 |
| **max_leaves** | 樹規模失控（葉子太多） | `breaker.max_leaves` (預設 1000) | 規劃期，每次子節點定案時 | 明顯壞掉的跳閘，不是精細調校目標 |
| **growth_stall_rounds** | 規劃期空轉（樹一直長，但葉子總數不收斂） | `breaker.growth_stall_rounds` (預設 6) | 規劃期，每輪循環開始時 | 邊長邊改，永遠拆不完 |
| **max_leaf_reflow** | 跨層垂直震盪（整合 ↔ 葉子反覆打回） | `breaker.max_leaf_reflow` (預設 3) | 執行期，整合失敗退回葉子時 | 葉子規格有問題，或底層實作與上層整合存在本質矛盾 |

### C-1. 整合輪不得「就地改葉子」繞過 `max_leaf_reflow`（防垂直震盪逃逸）
`max_leaf_reflow` 只在「整合失敗 → **宣告退回該葉子**（`tree_reflow_target` 填該葉子 + `reflow_count += 1`）」時推進。
若整合驗證輪不宣告退回、而是**當場直接 patch 葉子的程式碼**，`reflow_count` 永遠不漲，斷路器永遠不跳——
跨層垂直震盪就能無限繞圈而不被攔。故：
- 🚨 強制約束:**整合 / 中間節點的驗證輪,嚴禁直接修改任何葉子專屬檔案**(該葉子 output 範圍內的檔)。
  整合輪只能做整合驗證本身,以及修改「整合層自己的檔」(如 router 註冊、整合測試、`integration_contract`)。
- 🚨 葉子內容需要修 → **必須宣告 reflow**:把該葉子設 `NEEDS_REVISION`、`tree_reflow_target` 填該葉子 ID、
  `reflow_count += 1`,結束本輪交還引擎;下一輪才由葉子執行模型去修。這樣才會被 `max_leaf_reflow` 計數。
- ❌ 嚴禁:在整合輪「順手」改掉葉子程式碼讓整合過關。即使只是一兩行——那正是垂直震盪逃逸的典型手法,
  會被 git-review-gate 視為破壞性改動 REVERT(見其 §2「整合輪越界改葉子」)。

## D. 授權紅線
程式只被授權做「可機器判讀」的決定（如收斂計數、輪數、深度、樹規模、回流次數）。
凡牽涉 **價值判斷** 的決定，一律在邊界之外，程式不得擅自代判或放寬：
1. **「規格本身有矛盾」不該由程式判**：程式分不清「模型能力不夠」與「兩條需求本質衝突」，只需開 Issue 記錄現象，不可自行妥協或修改規格。
2. **「這棵樹拆得對不對」不該由程式判**：程式只能驗證「結構穩定」（集合收斂），正確性必須由人類 Gate 承接。
3. **「要不要放寬邊界繼續」絕不該由程式判**：撞到硬 BREAKER 時，唯一的合法動作就是 **停**。一個能自己放寬的邊界不是邊界，放寬只能由人類手動修改設定。

## E. 執行期「結構錯誤」上報的機械前提（防自抬逃生閥）
執行期 agent 可透過 `tree_structure_error=true`（見 state-model.md）宣告「缺葉子 / 需再拆」並交人。
這是逃生閥,不是把硬葉子甩給人類的出口——**比照 FROZEN,須有機械訊號才算數**：

- 🚨 強制約束:設 `tree_structure_error=true` 前,所引用的整合驗證項【必須】在 `last_round_result`
  歷史中**實際出現過 FAIL**(或該葉子 `reflow_count > 0`)。亦即「結構錯」必須是被客觀整合驗證
  打出來的,不能只是 agent 散文宣稱「我覺得這裡需要再拆」。
- 🚨 同時【必須】開一張 `BLOCKING` Issue,寫明:哪個整合驗證項 FAIL、缺哪個葉子 / 現有葉子為何
  拼不出需求、對照哪條 REQUIREMENTS。無此 Issue 佐證的宣告視為無效。
- ❌ 嚴禁:把「這片葉子難做 / 我這輪做不出來」當「結構缺漏」上報來開溜。**難 ≠ 結構錯**:
  難 → 繼續做,或讓引擎走升級階梯(fast→normal→thinking)換更強模型;只有「客觀缺葉子 /
  整合驗證證明現有葉子本質拼不出需求」才是結構錯。沒有整合驗證 FAIL 訊號 → 就是繼續做。
