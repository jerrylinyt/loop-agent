# 🧭 Loop Engineering Agent 設計藍圖（最新版 v2）

> **這份文件是什麼**:規劃「由 AI agent 反覆執行、直到收斂達標」的工程專案時的**設計原則與樣板**。
> **怎麼用**:把這份藍圖 + 你的新專案需求交給某個 AI agent，請它依本藍圖產出一套**可執行的主控文件**（CONTROL + 分階段檔 + 觸發器），給 Loop Agent 反覆讀取執行。
> **適用**:任何「單次做不完 / 不可信、需要中斷續做、需要多輪驗證收斂」的任務（legacy 逆向與重寫、大型遷移、大規模文件生成、資料清洗管線…）。技術中立，不綁特定語言或框架。

---

# 第 0 部：一句話總結 + 三段式生命週期

> Loop Engineering = 把一個大到單次做不完、也信不過單次結果的任務，拆成「**一份可持續推進的主控文件** + **反覆觸發的無狀態 agent** + **多層收斂驗證** + **震盪偵測與升級** + **Git 安全網**」，讓任何 agent 中斷、換 context、換模型後，都能只靠這份文件接手，做到**對齊目標且全部收斂**才停。

整體流程分三段（前兩段有人類確認點，第三段全自動直到收斂或需人類裁決）：

```
① 需求討論（互動）         產出 REQUIREMENTS.md（goal / DoD / 限制 / 驗收清單 / 需求編號）
       ↓ 〔人類最終確認需求〕
② 產生 Loop Plan           依需求產出 CONTROL.md + 分階段檔（任務、收斂、驗證、停止條件）
       ↓ 〔Plan Review Gate：需求全覆蓋？任務夠小？停止可判讀？無循環依賴？〕
       ↓ 〔人類確認 plan + 看輪數估算〕
③ 執行迴圈（自動）         反覆觸發 agent → 收斂 → 自我修正 → 直到達標
       ↑ 〔卡住/震盪偵測、模型升級、需求變更回流〕
       ↓
驗收：逐條檢查每條需求是否達成 →〔人類最終驗收〕→ DONE
```

# 第 1 部：核心問題與設計目標

Loop Agent 要同時解決六個現實問題：

1. **單次做不完**：超過單一 agent 的 context/時間上限 → 可中斷續做。
2. **單次信不過**：會漏看、腦補、判斷錯 → 重複驗證直到收斂。
3. **會中斷**：context 不足、斷線、換模型 → 狀態必須落在文件，不能只在 context。
4. **會把檔案改壞**：偶爾把檔案寫成空白/亂碼 → 可還原。
5. **會卡死**：改 A 壞 B、改 B 壞 A 的震盪，或同一問題一直修不好 → 要偵測得出、能升級、有逃生門。
6. **不知何時該停**：停止條件要**機器可判讀**、**不會死結**、**對齊目標**。

目標：**對齊目標、可接手、可驗證、可恢復、可收斂、不卡死、能自動停止。**

# 第 2 部：架構原則（九條）

## 原則 1 — 目標是一等公民（Goal first & traceable）
把 **goal 與完成定義（DoD）寫進文件**，並建立**需求 → 任務 → 驗證**的追溯：每條需求都對應到任務，最終驗證**逐條檢查每條需求是否被滿足**。
> 防「規格完整且自洽，卻解錯了問題」——內部指標全綠不等於達成目標。

## 原則 2 — 文件即狀態（State lives in the document）
所有進度、計數器、覆蓋率、待辦、Issue 都寫在文件裡，不靠 agent 記憶。
判準：把現在的 agent 關掉換一個新的，新 agent 能不能只靠文件接手？不能就是沒落地。

## 原則 3 — 單一進入點 + 分檔（Single entry + on-demand split）
一份 **CONTROL（主控檔）**＝規則 + 狀態表 + 計數器 + 索引，**每輪必讀**；數份**分階段/模組檔**＝詳細任務規格與驗證清單，**只讀當輪需要的那一節**。狀態放 CONTROL（單一事實來源），「怎麼做」放分檔。

## 原則 4 — 分階段 + 階段門（Phases + Phase Gate / 樹狀解鎖）
切成有依賴順序的階段（典型：分析/規劃 → 實作/產出）。階段間設 **Phase Gate**：前一階段「完全收斂」才放行下一階段，防「規格沒穩就亂做」。

**樹模式（漸進拆解特化）**：平 Phase Gate 升維成**樹狀解鎖**——父節點在**所有子節點 CONVERGED** 後才解鎖。整合驗證失敗時，可沿樹向下回流退回具體葉子重修（見 §3.10）。

## 原則 5 — 每輪只做一件事（One unit of work per round）
🚨 **強制約束（硬規則，非建議）**：agent 每次被喚醒，**只能且僅能**推進【單一一個】任務或執行【單一一次】驗證，做完落地、更新狀態後，**立即停止輸出、結束本 process，把控制權交還外部迴圈**。
❌ **嚴禁**：在同一輪內合併處理多個任務、批次做完多個 TODO、過了 Phase Gate 後又順手開新階段任務、或交棒後自行續跑下一輪。即使任務看起來很小、很快——也必須一輪一個、做完就停。
> 穩定交棒 > 一次衝完。agent 只是「外部迴圈呼叫一次的函式」，不是那個迴圈本身。

## 原則 6 — 不猜測（No guessing → Issue）
遇到不確定（未知格式、不明錯誤碼、無法解析的動態邏輯、缺檔）一律建 **Issue**，不准腦補。

## 原則 7 — 可追溯（Traceability，雙向）
分析階段每個結論附**來源**（檔名+行號）；實作階段每個產出檔標明**實作自哪份規格**。禁止「規格沒寫卻憑空產出」。

## 原則 8 — Git 是安全網（Git as recovery net）
每輪一個 commit ＝還原點；開機先自檢、檔案被寫壞就還原。

## 原則 9 — 卡死要有逃生門 + 授權紅線（Stuck → escalate → human）
偵測震盪/卡住 → 升級模型 → 仍不行就凍結互卡任務、交人類。**逃生門比任何自動修復都重要**，否則遇到「規格矛盾」會無限迴圈。

**授權紅線**：程式只被授權做「可機器判讀」的決定（收斂計數、輪數、深度、樹規模、回流次數）。凡牽涉**價值判斷**，一律邊界外、撞到就停、交人：
- **「規格本身有矛盾」不該由程式判**：程式分不清「模型能力不夠」與「兩條需求本質衝突」。
- **「這棵樹拆得對不對」不該由程式判**：程式能驗「穩定」，但穩定 ≠ 正確。正確性由人類 gate 承接。
- **「要不要放寬邊界繼續」絕不該由程式判**：撞到硬 BREAKER 時程式唯一能做的是停；放寬只能人下。**一個能自己放寬的邊界不是邊界。**

# 第 3 部：必備機制（可直接抄進新專案）

## 3.1 Agent 開機程序（Boot Sequence 樣板）
```
STEP G ▶ Git 完整性自檢：主控/關鍵檔是否被寫壞（空白/缺標題）→ git checkout HEAD -- <檔> 還原
STEP 0 ▶ 只讀 CONTROL 全文（不先讀分檔/原始資料）
STEP 1 ▶ 讀計數器與當前階段
STEP 2 ▶ 檢查停止條件（含 human_required）→ 成立就輸出訊號並結束
STEP 3 ▶ 檢查階段門 → 條件滿足就推進下一階段
STEP 4 ▶ 從狀態表挑「第一個未完成且非 FROZEN」的【唯一一個】任務（決定初稿/重驗/修正/驗證）
         🚨 一輪一任務：挑到第一個就停止挑選；❌ 嚴禁一次挑多個或批次處理
STEP 5 ▶ 只開「對應分檔的那一節」讀本輪任務規格
STEP 6 ▶ 只讀該任務宣告依賴的少數檔
STEP 7 ▶ 執行（寫檔守則：局部編輯優先、寫完讀回確認、寫壞且未 commit 就 checkout 還原）
STEP 8 ▶ 產出落地；發現前人錯誤 → 修正 + 寫修正記錄
STEP 9 ▶ 回 CONTROL 更新：狀態表、覆蓋率、Issue、計數器，
         並回填震盪偵測欄位 last_round_mode / last_round_result / last_round_fail_tasks
STEP C ▶ git add -A && git commit（本輪還原點）
STEP 10 ▶ 寫一筆執行日誌；🚨【立即停止輸出、結束本 process（Stop/Exit）】，控制權交還外部迴圈，
         由它喚醒另一個全新無狀態 agent 跑下一輪。❌ 嚴禁自行續跑下一輪或多輸出總結
```

## 3.2 任務生命週期與狀態
```
TODO → DRAFTED（初稿）→（多輪獨立重驗一致）→ CONVERGED（定稿）
                          ↑___ 發現實質差異就修正 + 收斂計數歸零 ___|
NEEDS_REVISION：被後續驗證打回需重做
FROZEN：因規格矛盾暫凍結，任務挑選時跳過，待人類解凍（見 3.7）
```

## 3.3 三層收斂（「信不過單次」的核心）
| 層級 | 防什麼 | 收斂判準 | 典型門檻 |
|------|--------|----------|----------|
| **單任務收斂** | 單支結果腦補/判斷錯 | 獨立重推一份 → 與既有比對，連續 N 次一致 | 2~3 |
| **集合穩定性收斂** | 大範圍**漏看** | 每輪獨立重新「列舉」項目，連續 N 輪集合不再增減 | 3~ |
| **全量驗證收斂** | 跨文件/模組**不一致** | 對整個產物全量驗證，連續 N 次 PASS | 10 |

**收斂鐵則 — 獨立性**：
- 重驗由「新一輪 agent（新 context）」做。
- **先重推/重列舉，再跟既有比對**——不可先讀既有結果再說「看起來沒問題」（橡皮圖章，同一盲點被一再確認）。
- 任一次發現實質差異 → 修正 + 收斂計數**歸零重數**。保證「連續 N 次」是對已穩定產物的 N 次獨立確認。
- **留痕才算數**：每次重驗/全量驗證都要留可稽核證據檔——重推稿每項附原始輸入來源（檔:行）以證明非抄既有產出；全量驗證附逐項 檔:行 重核結果與 build/test 原始輸出。**無證據檔的那一輪不得 +1**（細節見 convergence.md / completeness.md / boot-sequence.md STEP 4）。
- **收斂類別不准執行時自判**：一個任務算「需收斂 / Init 免收斂 / 有客觀把關」由規格預先宣告，執行 agent 不得臨場改判跳過重驗；有疑義從嚴歸「需收斂」。

## 3.4 覆蓋率要有明確分母
每個覆蓋率指標都定義清楚**分母來源**（「X 覆蓋率 = 已分析數 / 清單總數」）。分母模糊 → 各 agent 各自解讀 →「100%」失去意義。高風險專案再加「清單覆蓋」「原始資料行覆蓋」兩個防漏指標。

## 3.5 大範圍輸入的防漏協定（Completeness）
1. **列舉優先**：分析前先機械式掃描，把所有該分析單位（函數/事件/呼叫/分支/狀態…）含位置全列成**清單（分母）**，每項標 PENDING。
2. **分段精讀**：大檔分段讀，維護「閱讀涵蓋記錄」，確保每一行範圍被讀過、無跳段。
3. **逐單位深掘到分支層級**：每個函數的**每個分支都要交代**，不可只寫 happy path。
4. **閉合檢查**：每個被引用的東西都找得到定義、每個 handler/callback 都被分析；找不到 → 開 Issue。
5. **完整性 Gate**：清單無 PENDING、行覆蓋 100%、逐分支完成、閉合檢查過 → 沒過不准 CONVERGED。

## 3.6 Issue 分級（避免停止死結）
| 等級 | 定義 | 阻擋停止 |
|------|------|----------|
| `BLOCKING` | 不解決就無法正確完成 | ✅ 必須為 0 |
| `NON_BLOCKING` | 已知風險/外部依賴，可帶著完成 | ❌ 允許留存 |
> ⚠️ **千萬別把停止條件設成「Open Issue = 0」**——外部依賴的 Issue 永遠關不掉，會死結。改成「無 BLOCKING Issue」。
> 🚨 分級從嚴：有疑義一律先算 `BLOCKING`；❌ 嚴禁為了過停止條件（`blocking==0`）或避免 `consecutive_pass` 歸零，把真 BLOCKING 降級成 NON_BLOCKING（見 issues.md）。

## 3.7 震盪偵測 + 三模型二維調度 + 硬 BREAKER（防卡死與失控）★本版重點

### 3.7.1 三種卡住與訊號
| 卡法 | 現象 | 訊號 |
|------|------|------|
| 不收斂 | 同一問題一直修不好 | `rounds_since_progress >= stall_threshold` |
| **震盪 A↔B** | 每輪都「修好一個」卻原地繞 | 最近視窗的「失敗指紋」只在 ≤N 種間循環且有重複 |
| **無活動** | agent 連續多輪沒提交 | 活動簽章連續不變 |

**失敗指紋** = `hash( 排序(本輪失敗任務) + 排序(本輪改動檔案) )`，只在「失敗的驗證輪」推進歷史。
> 偵測放在**外部 loop 程式**（它每輪 parse CONTROL + 看 git 改動，最客觀；agent 繞圈時往往不自知）。

### 3.7.2 三模型角色 × 升級的二維調度
三個模型：**快速(fast) / 普通(normal) / 思考(thinking)**，同一組模型一魚兩吃：

**角色維（順風時按任務類型指派）**：
| 角色 | 模型 | 典型場景 |
|------|------|----------|
| 拆解/分析 | thinking | 規劃期需全局視野 |
| review/整合驗證 | normal | 獨立重推再比對 |
| 葉子執行 | fast | 執行期最小工作單位 |

**升級維（卡住時往上爬）**：`快速 → 普通 → 思考 → 人類`，沿既有 `stuck_level` 階梯。有進展則換回角色預設模型。

> 不是兩套模型系統：順風按角色、卡住當階梯，是同一組模型的二維調度。

### 3.7.3 升級階梯
```
Lv0 角色預設模型：偵測到震盪/卡住 → 升 Lv1
Lv1 升級模型：注入「先判根因」提示；有進展→換回角色預設；試滿 M 輪仍卡→升 Lv2
Lv2 最高模型(thinking)：開 BLOCKING Issue + FROZEN；再撐 K 輪無進展 → 停下交人類
```

### 3.7.4 三條硬 BREAKER（撞線即凍結交人，程式不准自我放寬）
| Breaker | 攔什麼失控 | config 旋鈕 | 預設 | 撞線代表 |
|---------|-----------|-------------|------|----------|
| **max_depth / max_leaves** | 無限遞迴拆解 | `breaker.max_depth` / `breaker.max_leaves` | 5 / 1000 | 最小單位定義訂錯，或此任務不該全自動 |
| **max_leaf_reflow** | 跨層垂直震盪（整合 ↔ 葉子反覆打回） | `breaker.max_leaf_reflow` | 3 | 該葉子規格可能本質有問題 |
| **growth_stall_rounds** | 樹一直動卻收斂不到全葉子 | `breaker.growth_stall_rounds` | 6 | 邊長邊改、永遠長不完 |

> ⚠️ `max_leaves` 是「明顯壞掉才觸發的跳閘」，不是把樹壓小的調校目標。
> 水平的 A↔B 震盪既有機制已會抓；**跨層的垂直震盪（max_leaf_reflow）是樹形才有的新形態，需新劃一條線。**

**關鍵設計**：
- 換模型只治「能力不足」，治不了「規格矛盾」。升級模型那輪**務必注入提示**：先判根因。
- **逃生門（FROZEN + human_required）比換模型更重要**。
- **BREAKER 不可自我放寬**：一個能自己放寬的邊界不是邊界。放寬只能由人類決定。

## 3.8 Git 安全網（防檔案被改壞）
- **一輪一 commit**：每個 commit 都是乾淨還原點，訊息含 Round/階段/任務。
- **開機自檢還原**：下一輪先驗證主控檔健康（非空、有標題），壞了 `git checkout HEAD -- <檔>` 救回。
- **寫檔守則**：局部編輯優先、寫完讀回確認、未 commit 的壞改用 `git checkout -- <檔>` 還原。
- **安全紅線**：禁止 `git reset --hard`、`git clean -fd`、`git checkout .`、`push --force`；只允許逐檔還原。
- `.gitignore` 忽略相依與建置產物，其餘進版控。

## 3.9 重複輸入用 Base + 變體
多個輸入近似複製時：先**完整**做一份**標準版（base）**，其餘只記/做**差異（diff）**，最大化複用，禁止整支複製再改。

## 3.10 樹形流程（漸進拆解特化）

> 樹是 **opt-in**：不啟用樹的既有平 phase 專案，行為零變動。

### 核心概念：拆解＝驗證同構樹
拆解樹與驗證金字塔是**同一棵樹**：
| 樹的位置 | 拆解語意 | 驗證語意 | 由誰做 |
|----------|----------|----------|--------|
| **葉子** | 最小工作單位（單一 component） | 單元驗證：component 自身可運行 | 快速（執行）→ 普通（review） |
| **中間節點** | 一組子項的整合 | 整合驗證：子項兜起來行為正確 | 視層級指派 |
| **根** | 整個任務 | 端到端驗證：對齊 REQUIREMENTS | 思考 / 人類 |

### 最小工作單位
> 快速模型能在一輪乾淨的短 context 裡可靠做對、而且能被獨立驗證的最小一塊。
> 具體形狀錨在 component：**新增單一 component 且其自身可運行**。

Proxy 圍欄（可校準）：`≤ max_files 檔、≤ max_lines 行、只碰一個關注點`。只有葉子受此約束。

### 規劃期（路線 A：生長只發生在規劃期）
```
② 規劃期：樹漸進生長 → 每輪只拆一個節點 → 集合穩定收斂
       ↓ 〔GATE：人類驗收整棵樹，點頭才放行〕
③ 執行期：回到靜態葉子清單，不再新增節點
```
- 每個節點的拆解用既有集合穩定機制收斂（重列子項、連續 N 輪不增減）。
- ⚠️ 集合穩定 ≠ 正確：穩定只代表弱模型沒梗了，正確性由人類 gate 承接。

### 人類 GATE（規劃→執行門）
- 收斂由程式判、正確性由人類判，**兩者不混**。
- Fail-path = **局部重拆**：人類標出有問題的子樹 → 只對該子樹重拆 → 重過 gate。

### 執行期：樹狀解鎖 + 回流邊
- **向上解鎖**：整合節點在所有子節點 CONVERGED 後才解鎖。
- **回流邊分兩種**：
  - **(a) 葉子內容錯** → 退回修復（受 `max_leaf_reflow` 管）。
  - **(b) 結構錯（缺葉子/需再拆）** → 停、交人（授權紅線：執行期任何結構變動一律交人）。
- **節點內收斂鐵則**：清單清空後仍須一次乾淨獨立重驗才算 CONVERGED。

### 節點狀態
`PENDING → DECOMPOSED → LEAF → IN_PROGRESS → CONVERGED`
`NEEDS_REVISION`（被回流退回）、`FROZEN`（breaker 撞線凍結）。

# 第 4 部：停止條件設計

好的停止條件要**機器可判讀**、**不死結**、**只在穩定且對齊目標時成立**、且**有人類逃生門**。

```
正常完成 = 最後階段全部 CONVERGED
        AND 全量驗證 consecutive_pass >= N（如 10）
        AND blocking_issues == 0
        AND 逐條需求驗收通過（原則 1）
        AND stop_condition_met == true

交人類停止 = human_required == true        # 卡死已凍結互卡任務，停下等裁決
```
要點：
- 計數器用 yaml 風格放 CONTROL，讓外部程式可 grep/parse。
- 任一次驗證 FAIL 或新增 BLOCKING Issue → `consecutive_pass` 歸零。
- 設 `max_rounds` 安全上限。
- 完成輸出明確訊號（如 `LOOP COMPLETE`）；交人類輸出明確警示。

# 第 5 部：外部觸發器（讓 Loop 真的轉起來）

## 5.1 觸發方式
外部程式（bash for-loop 或 Python）反覆呼叫 agent，每次丟**同一句話**：「讀 CONTROL，依 BOOT SEQUENCE 工作」。agent 自己從 `current_phase`/狀態表判斷該做什麼，觸發語不需每輪改。

## 5.2 外部程式要負責的四件事
1. **停止偵測**：每輪 parse CONTROL 計數器，判斷正常完成或 `human_required` 就停。
2. **震盪偵測 + 模型升級（3.7）**：維護失敗指紋環狀歷史，偵測震盪/卡住，依三層階梯切換模型指令（`current_model_tier`），過關後換回預設。
3. **Git 守護（外部第二層）**：每輪結束再檢查——主控檔被寫白就還原、agent 忘 commit 就補 autocommit。
4. **單一 append log**：詳細輸出 append 到一個 log（`tail -f` 看），終端機只留心跳。

## 5.3 兩個務必處理的工程坑
- **輸出緩衝**：很多 CLI 偵測到 stdout 非 TTY 就整批緩衝，導致「像卡住」。解法：讓 agent 繼承終端機，或用 **pty**（Python `pty.fork()` / bash `script`）配假終端機，兼顧串流與存檔。
- **換模型指令**：把模型參數做成可設定（`DEFAULT_MODEL` / `ENHANCED_MODEL` + 一個 `build_cmd()`），用你實際的 CLI 換模型寫法填入。

# 第 6 部：常見反模式（踩過的教訓）

| 反模式 | 後果 | 正解 |
|--------|------|------|
| 狀態只存在 agent context | 中斷就重來 | 狀態寫進文件 |
| 單一巨檔全塞一起 | 每輪 context 爆 | 主控檔 + 按需讀分檔 |
| 停止條件「Open Issue = 0」 | 外部依賴死結 | 「無 BLOCKING Issue」 |
| 只看內部指標、無 goal 追溯 | 規格自洽卻解錯問題 | 需求→任務→驗證逐條追溯 |
| 收斂時先讀既有再確認 | 橡皮圖章 | 先獨立重推再比對 |
| 大範圍讀一遍就寫 | 默默漏看 | 列舉優先 + 行覆蓋 + 逐分支 |
| 覆蓋率沒定義分母 | 各自解讀 | 每指標明確分母 |
| 相信「能編譯就對」 | 能跑但行為錯 | 行為對照規格逐項驗 |
| **改A壞B 無偵測** | **永遠卡 verify** | **失敗指紋偵測震盪 + 三層升級 + FROZEN 逃生門** |
| **卡住只會無腦換模型** | **規格矛盾燒光模型還在繞** | **增強模型先判根因；矛盾就開 BLOCKING 交人類** |
| agent 整檔重寫 | 寫成空白 | 局部編輯 + 讀回確認 + git 還原 |
| Python `capture_output=True` | 看不到即時輸出 | 繼承 TTY / pty 串流 |
| 直接寫程式不先定規格 | 規格漂移難驗 | 文件優先 + Phase Gate |

- [ ] 停止條件怎麼寫成機器可判讀的計數器？人類逃生門怎麼設？

**F. 安全與運維**

- [ ] 估算總輪數（收斂門檻 × 任務數 × 階段 × 驗證次數），決定要不要調門檻。

# 第 9 部：人類介入點現況與待議缺口（盤點）

> 本節是**現況盤點**，非設計定論。把散在引擎各處的人類 gate 集中列出，方便判斷「哪裡自動、哪裡停下交人、哪裡該停卻沒停」。待議缺口**尚未實作**，留待後續討論決定要不要補。

## 現有人類介入點（5 處）
| # | gate | 觸發 | 模式 | 位置 |
|---|------|------|------|------|
| 1 | 確認 REQUIREMENTS | 跑 `run.py` 之前，人類手動確認需求 | 全模式 | 流程上游（手動） |
| 2 | 規劃書 review | plan_loop 收斂 → 停 → 人類 review `.loop/` 後跑 `--stage execute` | **僅 gated** | `run.py`（gated 分支） |
| 3 | `plan_human_required` | 規劃迴圈卡死（升級模型仍無進展）→ 停下交人 | 全模式 | `plan_loop.py` |
| 4 | 樹「規劃期→執行期」gate | 全樹拆解收斂 → 人類驗收整棵樹才放行執行 | **僅 gated** | `plan_loop.py` break 後由 `run.py` 決定 |
| 5 | 執行期 `human_required` | Lv2 卡死 / FATAL_STATE 狀態檔毀損 / 樹結構錯 / 硬 breaker 撞線 | 全模式 | `loop.py` 多處 |

## 待議缺口（盤點，未實作）
- **A. 平模式階段轉換全自動**：平模式 analysis→implementation 的 Phase Gate（boot STEP 3）無人類確認點；只有樹模式有規劃→執行的人類 gate。高風險專案可能想要平模式也有階段確認點。
- **B. 無強制最終驗收 gate**：`LOOP COMPLETE` 直接結束 process；原則 1 要求的「逐條需求人類驗收」未被強制成終止狀態（機器判定 stop_condition_met ≠ 人類確認 DoD 達成）。
- **C. auto 模式略過樹的人類 gate**：auto 模式規劃收斂後直接接執行，略過 gate#4，與「集合穩定 ≠ 正確、正確性由人類 gate 承接」（原則 9 / §3.10）矛盾。
- **D. 中途改需求無 re-entry gate**：執行途中人類想修改 REQUIREMENTS 時，沒有定義好的回流/重新規劃介入流程。
- **E. NON_BLOCKING Issue 無驗收點**：可帶著一堆 NON_BLOCKING Issue 完成，無強制人類過目點（停止條件只看 BLOCKING==0）。

> 取捨提醒：人類 gate 越多越安全但越慢、越違背「自動跑到收斂」的初衷。每個缺口要不要補，取決於專案風險檔次——**這是價值判斷，留給人類決定，程式不自行加擋。**

---

> **總結心法**：把「不可信、做不完、會中斷、會卡死」的大任務，轉成「一份會自己記錄進度與信心、能自我修正、卡住會升級、能自動判斷何時收斂或求救」的活文件；agent 只是反覆來推它一把的手。**對齊目標、穩定、可追溯、不猜測、能還原、卡死有逃生門**，比跑得快重要。
