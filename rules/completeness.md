# 🔬 RULE — 大範圍防漏協定（Completeness）

> **唯讀框架規則**。大範圍輸入(大檔、多函數、多事件、多項目)最容易「讀一遍就漏」。
> 對策:**先機械式列舉建清單(分母),再強制每一項都被交代到,並追蹤輸入覆蓋率。**
> 適用於任何「大範圍怕漏」的任務。門檻 `page_converge`(或對應名稱)來自 `loop.config.yaml.extra_thresholds`。
> 領域範例(下方 A 的清單)以程式碼為例,**請依你的輸入語言/格式調整**。

## A. 列舉優先（先建清單,再深掘）
深入分析「之前」,先對全部來源做一次**機械式掃描**,把所有「該分析單位」含位置(檔:行)全列出,寫進一份 `inventory`(清單檔)。每項標一個狀態欄:`PENDING` / `ANALYZED` / `DEAD`。
這份清單就是「**分母**」。
> 🚨 `DEAD`(確認未使用)是高風險逃生閥,【必須附機械式證據】,不接受純文字理由:例如貼出「grep/搜尋該符號全專案零引用」的結果,或指出它被哪個已分析單位明確排除。❌ 嚴禁為了清空 PENDING、讓完整性 Gate 過關,把「難分析」的單位草率標成 DEAD。有疑義 → 保持 PENDING 或開 Issue,不得標 DEAD。

> 程式碼範例的列舉項(依語言調整):函數定義、事件綁定、網路呼叫(ajax/fetch/RPC)、彈窗/對話框、表單與驗證、每個函數內的重要分支(if/else/switch/early-return/try-catch)、全域狀態/旗標、動態危險物(eval/字串拼接 selector/HTML)。
> 非程式碼任務的列舉項:資料欄位、記錄類型、狀態轉換、規則條目、章節…依任務本質定義。

## B. 分段精讀（大檔不跳讀）
- 來源檔過大(如 >300 行)→ **分段讀**(每段 ≤200 行),不要只抓開頭結尾。
- 在 inventory 底部維護「**閱讀涵蓋記錄**」:已精讀的行範圍(如 `src/a.js 1-200 ✓ / 201-400 ✓`),確保整檔每一行被涵蓋、無跳段。

## C. 逐單位深掘（分支層級完整性）
對 inventory 裡每個單位,在對應產出檔記:被誰呼叫/觸發時機、輸入/讀取的狀態、**每個分支分別做什麼(逐分支列,不可只寫 happy path)**、副作用、觸發的呼叫/開的對話框/導頁、呼叫了哪些其他單位。

## D. 閉合檢查（cross-reference closure）
- 每個「被呼叫的單位」都要能在 inventory 找到定義(找不到 → 缺檔/在別處 → 開 Issue)。
- 每個「事件綁定」都要有對應 handler 被分析。
- 每個「呼叫的 success/error callback」都要被分析。

## E. 完整性 Gate（沒過不得 CONVERGED）
```
□ inventory 所有項目狀態都是 ANALYZED 或 DEAD（無 PENDING）；每個 DEAD 都附了機械式零引用證據（缺證據視同未過）
□ 閱讀涵蓋記錄顯示整檔行範圍 100% 被精讀
□ 每個單位都有逐分支說明（不是只有 happy path）
□ D 的閉合檢查全部通過（無懸空呼叫/無未分析 handler）
```

## F. 集合穩定性收斂迴圈（比單任務收斂更嚴,治大檔漏看）
> 對「怕漏」的重災區任務,用「**項目集合穩定性**」當收斂依據:
> **連續 `page_converge_threshold` 輪「獨立重新列舉」都列不出新項目,集合才算穩定。**

每一輪做這類任務時(一輪仍只處理這【單一一個】列舉任務、做一次獨立重列舉,不批次掃多個任務),在 inventory 底部維護「收斂記錄」表(Pass / Round / 本輪獨立列舉項目數 / 與上輪比較 / converge):
```
每一輪（獨立列舉,先不看舊 inventory）:
  1. 從來源重新做一次機械式列舉,得到本輪「項目集合」。
     🚨 防橡皮圖章(記錄標準對齊 convergence.md step 1/5):本步驟【必須】先「不看舊 inventory」,
        把本輪獨立列舉結果(每項含 檔:行)寫成 scratch 獨立驗證檔(如 <outputs>/.enum/<task>-R###.md),
        再進第 2 步比對。❌ 嚴禁先打開舊 inventory 看一眼就宣稱「本輪無新項」→ converge+1;
        無驗證檔的那一輪集合比對無效,不得 converge+1。
  2. 與既有 inventory 做集合比較（比「項目身分」,非逐字):
       · 有「新項目」(本輪列到、舊的沒有) → 漏看!補進 inventory + 補分析 + converge=0（歸零）。
       · 有「該移除項目」(舊的誤列/不存在) → 修正 + converge=0。
       · 集合完全一致（無增無減） → converge += 1。
  3. 在收斂記錄新增一列。
  4. converge >= page_converge_threshold 且通過 E 的完整性 Gate → 才可 CONVERGED。
```

## 覆蓋率要有明確分母（見 BLUEPRINT 3.4）
每個覆蓋率指標都定義清楚**分母來源**(如「X 覆蓋率 = 已分析數 / 清單總數」)。分母模糊 → 各 agent 各自解讀 →「100%」失去意義。

## Context 防爆連動（見 context-budget.md）
- 完整性靠**列舉清單 + 行覆蓋追蹤**,**不靠「把檔案再讀一遍確認」**。
- inventory 只放「清單(分母)+ 狀態欄 + 行覆蓋記錄」,**不放分析內文**(分析內文進對應產出檔,一檔一主題)。
